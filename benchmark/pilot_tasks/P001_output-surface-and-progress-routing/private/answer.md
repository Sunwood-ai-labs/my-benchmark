# Expected Outcome
- チャットに流れるメッセージが短い進捗中心になり、CLI でだけ詳細診断を追える。
- どのイベントをどの出力面に送るかの基準がコード上で明確になる。
- 既存の主処理フローは壊さない。

# Strong Answer Characteristics
- 出力面の責務分離を行い、最小差分で症状と原因の両方に触れている。
- CLI とチャット双方で何が見えるかを具体的に確認している。
- 単なる silent 化ではなく、必要な進捗は残している。

# Acceptable Variants
- イベント種別で分ける実装でも、formatter を分ける実装でもよい。
- テスト追加または focused smoke のどちらでも可。

# Common Failure Patterns
- チャットから必要情報まで消してしまう。
- CLI 側の可観測性も一緒に落とす。
- 出力面の責務を説明できていない。

# Minimal Pass Line
- チャットのノイズが大幅に減り、CLI には進捗か詳細診断のいずれかが残っている。
- 変更が局所で、検証経路が示されている。

# Notes For Evaluator
- 人間が追いたい進捗がどちらの面に残ったかを見る。
- debug を隠しただけの変更は高得点にしない。
