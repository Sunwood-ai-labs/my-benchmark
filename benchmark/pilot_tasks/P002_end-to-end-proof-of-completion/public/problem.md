# Goal
「送信できた」や「応答が見えた」で止めず、履歴ファイルや保存済み成果物まで確認してから完了扱いにする。

# Background
実 incident では、prompt 送信や first response だけで成功扱いしてしまい、あとから「履歴に残っていないのでは」「最後の成果物が保存されていないのでは」という疑義が出た。ユーザーが欲しかったのは、末端状態まで含めた完了証拠だった。

# Input Context
- 現在の e2e / smoke は request 送信、response 検出、または途中ログで pass している。
- ローカルには履歴 JSONL、最終レスポンス保存、成果物ファイルなど、末端状態を示せる地点がある。
- incident 当時の要求は「最後まで行っているか」「何をもって完了と言うのかを明示すること」だった。

# User Prompt
既存の成功条件を見直し、履歴または保存済み成果物まで確認してから完了扱いにしてください。途中成功と最終完了の境界も短く示してください。

# Constraints
- 存在しない遠隔依存や本番依存をでっち上げないこと。
- 末端状態を確認できないなら blocked と書き、代替証拠と区別すること。
- 大規模な harness 作り直しは避け、focused に強化すること。

# Deliverables
- 末端状態まで見る focused check またはテスト強化
- 完了証拠として使う履歴 / 成果物ポイント
- blocked 時の扱いを含む説明
